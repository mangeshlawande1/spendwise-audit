# Round 2 Reflection

---

## 1. Most uncomfortable trade-off due to time pressure

Storing `user_email` directly on the `audits` table instead of normalising it into a `users` table with proper foreign keys. The "right" schema has a `users` table, audits reference it, and the email is a single source of truth. What I shipped puts `user_email` as a denormalised column on `audits` — so if a user submits 3 audits, their email is stored 3 times. It also means I can't easily update all their records if they change email.

I made this call because a `users` table would require: a migration, a new `saveUser()` function, a join in every `getAuditsWithEmail()` query, and careful handling of the "email submitted after audit" race condition. Under 36 hours, the denormalised column gets the feature shipped with no regressions. I'd fix it in the next sprint and noted it as a risk in the PR.

## 2. If given 24 more hours, the first thing I'd do

Wire the Vercel Cron and test the full automated loop in production — not local dev. Right now `detect-changes` is a working endpoint that you call manually. The gap between "manually callable" and "actually runs on a schedule" is one `vercel.json` file and 20 minutes of testing — but it's the difference between a feature that works and a feature that does what it promises. I'd also send a real test email from a deployed domain instead of Resend's sandbox, because that's what reviewers will actually see when they click the link.

## 3. What my Round 1 self made harder for Round 2 self

The audit ID is generated by `nanoid()` inside `runAudit()`, which means the ID is set before the result is saved. That was fine in Round 1 (in-memory store, ID is immediately usable). In Round 2, when I need to link a re-audit back to its parent, I have to pass the `id` of the already-generated original audit as `parent_audit_id`. It works, but it means the ID generation and the DB save are separated — if the save fails, the ID is orphaned.

The cleaner Round 1 design would have been to generate the ID in the API route handler (after the DB confirms the insert), not inside the pure `runAudit()` function. I kept the existing pattern to avoid breaking changes, but it's the kind of coupling that bites you in iteration 2.
